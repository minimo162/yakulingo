# YakuLingo — M365 Copilot Translation App: Translation Result Display Fix

## Overview
YakuLingo is a bidirectional translation application built on M365 Copilot.
It translates Japanese to English (with multiple style options) and other
languages to Japanese (single result with explanation). Translation is
performed via M365 Copilot chat UI automated through Edge WebDriver (Playwright).

## Problem Statement
Three issues prevent the app from becoming operational:

### Issue 0: Startup Freeze at "準備中..." Screen (ACTIVE)
アプリ起動時に「準備中...」画面でフリーズし、翻訳可能状態に遷移しない。

**原因分析:**
GPTモード自動選択は既にno-op化済み（`ensure_gpt_mode()`, `ensure_gpt_mode_required()`,
`_ensure_gpt_mode_setup()` すべて即座にreturn）。

起動時のフローは以下の通り:
1. `connect()` → Edge起動 + CDP接続 + ログインチェック → 成功
2. `_wait_for_copilot_ready()` (app.py:2290) → `check_copilot_state()` をポーリング
   - タイムアウト: **3600秒（1時間）** — `RESIDENT_STARTUP_READY_TIMEOUT_SEC`
3. `check_copilot_state()` (copilot_handler.py:6347) → URLが/chatでも
   チャット入力欄セレクタが見つからないと `LOADING` を返す (line 6421-6429)
4. `wait_for_prompt_ready()` (copilot_handler.py:6804) → 入力欄の可視性チェック
   - タイムアウト: **300秒** — `RESIDENT_STARTUP_PROMPT_READY_TIMEOUT_SEC`

**フリーズの直接原因候補:**
- `_check_copilot_state()` line 6421-6429: `/chat` URLにいるが `CHAT_INPUT_SELECTOR_EXTENDED`
  が見つからず `LOADING` を返し続ける → ポーリングが最大1時間続く
- Copilotページの読み込みが遅い/完了しない
- `/landing` や `/landingv2` から `/chat` へのリダイレクトが行われない

**Copilotチャット入力欄の最新HTML (2026-03-31時点):**
```html
<span role="textbox" contenteditable="true" tabindex="0"
      aria-label="Copilot にメッセージを送信する"
      id="m365-chat-editor-target-element"
      class="fai-EditorInput__input ..."
      data-placeholder="Copilot にメッセージを送信する"
      data-lexical-editor="true">
```
→ セレクタ `#m365-chat-editor-target-element, [data-lexical-editor="true"]` はマッチするはず。
→ タイミング問題（DOM未描画）の可能性が高い。

- `ensure_gpt_mode()` → 即座にreturn
- `ensure_gpt_mode_required()` → `True` を即座にreturn

### Issue 1.5: New Chat Button Selector Update
Copilot UIの「新しいチャット」ボタンの要素が変更された可能性がある。

**現在のコード上のセレクタ** (`copilot_handler.py:1199`):


**最新のCopilot UIのボタン要素** (2026-03-31時点):

**変更点:**

- `data-automation-id="newChatButton"` が新規追加

現在のセレクタはOR条件(`data-testid` / `aria-label`)でマッチするため

### Issue 2: Translation Results Not Displayed
Translation results are not displayed in the app UI. Instead, only the
placeholder text "翻訳結果がここに表示されます" is shown. This may be a
downstream consequence of Issue 1 (translation never completes), or an
independent issue in the result capture/parse/display pipeline.

### Issue 4: 翻訳オーバーヘッドの微調整

Copilot翻訳はAI応答時間が支配的だが、送信〜応答取得のオーバーヘッドに
固定sleep等の無駄がある。体感差は小さい（0.3〜0.5秒程度の改善）が、
連続翻訳時に累積する。

**翻訳1回あたりのオーバーヘッド内訳（AI応答時間を除く）:**
```
new chat クリック     ~0.05s  （非同期化済み）
プロンプト入力        ~0.2-0.3s
送信ボタン待機        ~0.1-0.5s
送信確認ポーリング    ~0.2-0.8s
送信後安定化sleep     0.3s ×2箇所 ← 改善対象
応答安定性確認        ~0.3-1s
合計オーバーヘッド    ~1.2-3.0s
```

**改善箇所:**

**A. 送信後安定化sleepの短縮** (copilot_handler.py:9266, 9333)
- 現在: `time.sleep(0.3)` ×2箇所（送信確認後とlate verify後）
- 目的: Copilot内部状態の安定化（DOM操作中の「応答を処理中です」防止）
- 改善: 0.3s → 0.15s に短縮（計0.3s節約）
- リスク: 短すぎると「応答を処理中です」がDOMに混入する可能性

**B. Late verify タイムアウトの短縮** (copilot_handler.py:9295)
- 現在: `LATE_VERIFY_MAX = 1.5` 秒
- 目的: Enter送信後のフォールバック確認
- 改善: 1.5s → 1.0s に短縮
- リスク: 初回Enter送信の確認が取れずリトライが増える可能性

**C. check_copilot_state()のチャット入力チェック復元** (copilot_handler.py:6441-6448)
- 誤って/chatページでチャット入力の有無に関わらずREADYを返す変更が入っていた
- チャット入力がない場合はLOADINGを返すよう修正済み（意図しない変更の修正）

### Issue 5: テキスト翻訳のCopilot応答時間が長い（3スタイル比較）

「こんにちは」1語の翻訳に27秒かかっており実用的でない。
オーバーヘッド（1-3秒）ではなく、**Copilot AIの応答生成時間が支配的**。

**現状のフロー:**
```
translate_text_with_style_comparison()
  → 3スタイル比較プロンプト（60行）を一括送信
  → Copilotが3スタイル×(翻訳+日本語解説)を生成 → ~15-20秒
  → パース失敗時: _looks_untranslated_toリトライ → +15-20秒
  → それも失敗: 3スタイル個別翻訳フォールバック → +15-20秒×3
  最悪ケース: 27秒 × 4-5回 = 2分以上
```

**3スタイル比較は必須要件。**

**根本原因:**
1. プロンプトが長い（60行の指示 + 翻訳ルール）→ Copilotの処理時間増大
2. 出力要求が大きい（3スタイル×翻訳+詳細解説）→ 応答生成トークン数が多い
3. 短文でも長文でも同じプロンプト/出力量を要求
4. パース失敗時のリトライが重い（最悪4-5回のCopilot呼び出し）

**改善方針:**

**A. 解説の簡素化** (prompt_builder.py:284)
- 現在: 「Explain in Japanese with the same level of detail as an individual translation」
- 改善: 1-2文の簡潔な解説に変更
- 効果: Copilotの出力トークン数を約半減 → 応答時間~10秒短縮

**B. プロンプト圧縮** (prompt_builder.py:242-298)
- 現在: 60行のテンプレート（冗長な禁止事項の列挙）
- 改善: 30行程度に凝縮（意味を保ちつつ重複を統合）
- 効果: 入力トークン削減 → ~2-3秒短縮

**C. 短文用の軽量パス** (translation_service.py:2018付近)
- 短い入力（例: 50文字以下）の場合、解説を省略 or 3スタイルまとめて1つの解説にする
- 効果: 短文翻訳で劇的改善（~15秒短縮）

**D. リトライ抑制** (translation_service.py:2050-2102, 2143-2161)
- `_looks_untranslated_to_en` のリトライが過剰（最悪4回追加呼び出し）
- 3スタイル個別フォールバック（2147-2161行）は最終手段としてのみ発動
- パース失敗回数の上限を設け、早期にエラー返却

**E. ストリーミング表示の改善** (app.py:10088-10116)
- 現在: 応答全体完了後にパースして表示
- 改善: 1スタイル目の翻訳パースが完了した時点で即表示、残りを順次追加
- 効果: 体感速度の大幅向上（実質的な待ち時間は1スタイル分に）

### Issue 6: Copilotセッション・ウォームアップによる翻訳レイテンシ削減

翻訳リクエストのたびにCopilotのセッション初期化やネットワーク接続の
コールドスタートが発生し、体感待ち時間が長い。事前に翻訳プライミング
メッセージを送信してセッションを温めておくことで、実際の翻訳リクエスト時の
レイテンシを削減する。

**核心的な設計原則: ウォームアップ済みチャットを翻訳にそのまま再利用する**

現在の実装は翻訳開始時に `start_new_chat()` を呼ぶため、ウォームアップで
温めたチャットセッションが破棄されてしまい、ウォームアップの効果が無駄になる。

**修正前（効果なし）:**
```
warmup: 新チャット → プライミング送信 → 応答取得（セッション温まる）
翻訳:   新チャット ← ★セッションリセット、warmup無駄
        → 翻訳プロンプト送信 → 応答取得
```

**修正後（正しいフロー）:**
```
warmup: 新チャット → プライミング送信 → 応答取得（セッション温まる）
翻訳:   ★新チャットを開かない → そのまま翻訳プロンプト送信 → 応答取得
翻訳後: 新チャット → プライミング送信 → 応答取得（次回用warmup）
```

#### Phase A: 起動時ウォームアップ (実装済み、チャット再利用は未修正)
```
アプリ起動 → Early Edge起動 → connect() → warmup() → ユーザー操作待ち
```
- `_early_connect()` で `connect()` 成功後に `warmup()` を呼ぶ
- バックグラウンドスレッドで実行（UIをブロックしない）
- `warmup_on_connect` 設定でON/OFF可能（デフォルトON）

#### Phase B: 翻訳間ウォームアップ (実装済み、チャット再利用は未修正)
```
翻訳完了 → 新チャット → プライミング送信 → 応答取得 → 次の翻訳待ち
```
- `_translate_single_impl()` 完了後に `_start_background_warmup()` で非同期実行
- `_cancel_background_warmup()` でウォームアップ中断可能

#### Phase C: ウォームアップ済みチャットの再利用 (未実装 ← 次のタスク)

翻訳メソッドがウォームアップ済みチャットをそのまま使うように変更する。

**変更箇所:**

1. **ウォームアップ完了状態の追跡**:
   - `_warmup_chat_ready: bool` フラグを追加
   - `_warmup_impl()` が正常完了したら `True` にセット
   - `start_new_chat()` が呼ばれたら `False` にリセット

2. **`_translate_single_impl()` の変更**:
   - 翻訳開始時、`_warmup_chat_ready` が `True` なら `start_new_chat()` をスキップ
   - `_warmup_chat_ready` を `False` にリセット（再利用は1回のみ）
   - `_warmup_chat_ready` が `False`（ウォームアップ未完了 or 既に使用済み）なら
     従来通り `start_new_chat()` を呼ぶ
   - 翻訳完了後に `_start_background_warmup()` で次回用ウォームアップ開始
     （`_start_background_warmup()` 内で新チャット → プライミング → 応答取得）

3. **`_translate_sync_impl()` の変更**:
   - 同様に `_warmup_chat_ready` が `True` なら初回バッチで `start_new_chat()` をスキップ
   - 2回目以降のバッチは従来通り（同一チャット内で連続送信）

4. **キャンセル処理**:
   - ウォームアップ応答待ち中に翻訳リクエストが来た場合:
     `_cancel_background_warmup()` → ウォームアップ中断 → `_warmup_chat_ready = False`
     → 従来通り `start_new_chat()` で翻訳開始
   - ウォームアップが完了している場合: そのままチャットを再利用

5. **ウォームアップメッセージ**:
   翻訳プライミング用メッセージを使用（汎用挨拶ではなく翻訳コンテキスト設定）:
   ```
   あなたは翻訳のスペシャリストです。これから私の翻訳指示に従ってください。
   ```

**連続翻訳時のフロー:**
```
起動 → warmup（新チャット＋プライミング）→ chat_ready=True
  ↓
1回目翻訳: chat_ready=True → 新チャットスキップ → 翻訳送信 → 応答取得
  → chat_ready=False → background warmup開始（新チャット＋プライミング）
  ↓
2回目翻訳: warmup完了していれば chat_ready=True → 新チャットスキップ
           warmup未完了なら chat_ready=False → 新チャット → 翻訳送信
  ↓
（以降繰り返し）
```

**設定**: `warmup_on_connect` をPhase A/B/C すべてに適用（1つの設定で制御）

### Issue 7: ウォームアップ応答のUI漏れとDOM残留

Phase Cでウォームアップ済みチャットを再利用する際、ウォームアップの応答が
DOMに残ったまま翻訳プロンプトが送信されるため、以下の問題が発生する。

**症状:**
- ウォームアップ応答（プライミングメッセージへの返答）がUIにストリーミング表示される
- UIが予期しない状態になる（翻訳結果の代わりにウォームアップ応答が表示される）

**根本原因:**
ウォームアップ済みチャットを再利用する際に `start_new_chat()` をスキップするが、
そのチャットにはウォームアップの応答が残っている。翻訳の `_get_response()` が
ポーリングを開始すると、DOMから「最新の応答」を抽出するが、ウォームアップ
応答がまだDOMに存在するためそれを拾ってしまう。

**応答漏れの経路:**
```
warmup完了 → 応答がDOMに残留 → _warmup_chat_ready=True
  ↓
翻訳開始 → start_new_chat()スキップ → 翻訳プロンプト送信
  ↓
_get_response() → _get_latest_streaming_text() → DOMからウォームアップ応答を検出
  ↓
on_chunk(warmup応答テキスト) → UIストリーミングプレビューに表示
```

**影響箇所:**

1. `_warmup_impl()` (copilot_handler.py:3310):
   - `start_new_chat(click_only=True)` は `click_only=True` のため
     `_wait_for_responses_cleared()` を呼ばない
   - ウォームアップ応答がDOMに残る

2. `_translate_single_impl()` (copilot_handler.py:8226):
   - `_warmup_chat_ready=True` 時に `start_new_chat()` をスキップ
   - 旧応答のクリアなしに新プロンプトを送信

3. `_get_response()` (copilot_handler.py:~10147):
   - `_get_latest_streaming_text()` がDOM内の最新応答を返す
   - ウォームアップ応答がまだ「最新」の場合それを返す

4. `on_chunk` コールバック (app.py:~4059):
   - `_streaming_preview_label.set_text(partial_text)` でUIを更新
   - ウォームアップ応答テキストがUIに漏れる

**修正方針（検討経緯と採用結果）:**

**A. ウォームアップ後に応答マーカーを記録** → 不採用
   - DOM要素数ベースのマーカーは信頼性に不安

**B. ウォームアップ後に `start_new_chat()` で応答クリア** → 試行後却下
   - 実装して検証したが、新チャットを開くことでプライミングコンテキストが失われ、
     ウォームアップと翻訳が別チャットで実行される問題が発生した
   - ログ分析で確認: `start_new_chat` がチャットBを作成し、プライミングはチャットAに残る

**C. DOM要素数オフセット方式** → 不採用
   - DOM構造への依存が大きく脆弱

**D. start_new_chat後すぐ翻訳** → 試行後却下（Bと同様の問題）

**E. baseline_text方式（初期版）** → 不完全、修正必要
   - `_warmup_impl()` でウォームアップ応答テキストを `_warmup_baseline_text` に保存
   - `_get_response()` に `baseline_text` パラメータを追加
   - `last_text = baseline_text` で初期化 → 初回ポーリングでは漏れを防止
   - **問題**: DOMの `_get_latest_streaming_text()` はウォームアップ応答+翻訳応答を
     連結した全テキストを返す。翻訳テキストが追加されると
     `"warmup+translation" != "warmup"` で `on_chunk` が発火し、ウォームアップ
     テキスト込みでUIにストリーミングされる
   - **さらに**: `_get_response()` の最終返却値にもウォームアップ応答テキストが含まれる
     ため、パーサー (`_parse_style_comparison_result`) が期待するフォーマット
     (standard/concise/minimal セクション) と一致せず
     "Failed to parse style comparison result" エラーとなる

### Issue 8: baseline_text方式の不完全性 — ストリーミング漏れとパース失敗

**症状:**
1. ウォームアップ応答がUIにストリーミング表示される（baseline_textでブロックできない）
2. "Failed to parse style comparison result" エラーで翻訳結果を表示できない

**根本原因:**
DOMのテキスト抽出 (`_get_latest_streaming_text()` → `_get_latest_response_text_fast()`)
は同一チャット内の全応答テキストを連結して返す。baseline_text方式は `last_text` の
初期値をウォームアップ応答に設定するだけなので、翻訳テキストが追加されると
全体テキストが `last_text` と異なり、ウォームアップ応答ごと `on_chunk` に流れる。

```
ウォームアップ応答: "はい、翻訳のお手伝いをします。" (baseline_text)
  ↓
翻訳プロンプト送信 → Copilotが応答開始
  ↓
DOM全テキスト: "はい、翻訳のお手伝いをします。\n\n[standard]\n訳文: ..."
  ↓
current_text != last_text("はい、翻訳のお手伝いをします。") → True
  ↓
on_chunk("はい、翻訳のお手伝いをします。\n\n[standard]\n訳文: ...") → UI漏れ
  ↓
_get_response()最終返却値にもウォームアップ応答が含まれる
  ↓
パーサーが不正なプレフィックスのせいでスタイルセクションを正しく抽出できない
  ↓
"Failed to parse style comparison result"
```

**修正方針:**

`_get_response()` 内で `baseline_text` が設定されている場合、DOMから取得した
`current_text` からウォームアップ応答プレフィックスを除去（ストリップ）する:

1. ポーリングループ内で `current_text` が `baseline_text` で始まる場合、
   プレフィックスを除去してから `on_chunk` に渡す
2. 最終返却値からも同様にプレフィックスを除去する
3. これにより `on_chunk` には翻訳テキストのみが流れ、パーサーにも
   クリーンな翻訳結果のみが渡される

## Goals
1. Diagnose the exact point of failure in the translation → display pipeline
2. Fix the root cause so translation results are displayed correctly
3. Ensure both text translation and file translation display results
4. Maintain existing functionality (glossary, bilingual output, style options)

- Changing the Copilot-based translation engine
- Redesigning the UI
- Adding new features

```python
TextTranslationResult:
    source_text: str
```

- After a text translation request, the result panel shows the translated text
- For JP→EN: Three style options (standard, concise, minimal) are displayed
- For other→JP: Single translation with explanation is displayed
- Error states show meaningful error messages, not the empty placeholder
